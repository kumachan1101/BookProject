# おわりに
## 1. 原則は目的ではなく、価値を生むための手段である

本書を最後まで読み進めたエンジニアの皆様は、C言語によるソフトウェア開発のために多くの原則・構造・実装技法を習得しました。第1部で「道具」を手にし、第2部で「指針（SOLID）」を学び、それらを統合的に活用する術を身につけました。

最後に、本書が最も強調したいメッセージを記します。それは、 **「原則を守ること」自体がゴールになってはいけない** ということです。

## 2. 設計原則の「罠」に、はまらないために

真面目なエンジニアほど、「本に書いてあったから」とルールを厳格に適用しようとしがちです。しかし、目的を忘れた適用は、かえってコードを複雑にし、開発の足を引っ張ることになります。

### 2.1. 手段と目的の分離

同じ「原則の適用」でも、その動機によって結果が「良い設計」になるか「負債」になるかが分かれる分岐点です。常に「なぜやるのか（Why）」という **目的の確認** を問い続けることが、過剰設計（手段の目的化）を防ぐ唯一の方法です。

```mermaid
graph TB
      A["原則の適用"] --> B{"目的は?"}
      B -- "コストを下げるため" --> C["良い設計: 価値の創出"]
      B -- "ルールを守るため" --> D["過剰設計: 負債の創出"]

      subgraph "手段が目的化する罠"
          D --> E["複雑性の増大"]
          E --> F["理解コストの上昇"]
          F --> G["変更が困難になる"]
      end

      style C fill:#e8f5e9,stroke:#2e7d32
      style D fill:#ffebee,stroke:#c62828
```

 **良い設計** とは、単にルールを遵守することではなく、 **「明日の自分や仲間が、楽に変更できる状態を作ること」** です。その本来の目的を見失わないよう、常に「これは本当に必要か？」「これで読みやすくなるか？」を自らに問い続けてください。

## 3. 正解のない世界で「自分の軸」を持つ

設計という行為に、唯一無二の正解はありません。あるのは、その時々の状況における「最適解」と「トレードオフ」だけです。

本書で示したのは、あくまで一つの地図に過ぎません。これからは、あなた自身の現場での経験を積み上げ、 **「自分なりの設計の基準」**  を確立してください。

- 「今回のプロジェクトの規模感では、このレベルの抽象化が最もメンテナンスしやすかった」
- 「この依存の切り方が、チームにとって一番理解しやすかった」

そうした実体験に基づく「軸」を持つことが、設計者としての真の強みになります。 **本書の内容をそのまま守る必要はありません。**

本書を土台にして、あなた自身の設計哲学を築き上げてください。

## 4. 最後に ―― 設計という冒険を楽しもう

本書で紹介したあらゆる設計原則は、すべて **あなたを助けるための道具** です。道具に振り回されるのではなく、道具を使いこなし、素晴らしいソフトウェアを作り上げてください。

思い通りに、軽やかに変更できる設計ができたときの楽しさや嬉しさは、エンジニアにとって何物にも代えがたい報酬です。その楽しさを、ぜひ現場の最前線で体感してください。

あなたがこれから作るソフトウェアが、あなた自身と、それに関わる人々に喜びをもたらし、長く愛されるものになることを確信しています。設計を楽しみましょう。

あなたの挑戦を応援しています。

## 5. 著者について

**花島 弘明（Hiroaki Hanajima）**

ソフトウェア開発に携わって20年になりますが、設計の難しさは今でも変わりません。日々勉強の毎日で、自分の未熟さを痛感することばかりです。

昨今、AIの登場によってプログラミング自体は飛躍的に楽になりました。しかしだからこそ、生成されたコードが「適切な設計になっているか」を判断し、導く力がこれまで以上に求められていると感じています。オブジェクト指向ではないC言語であえて設計を取り上げたのは、言語の機能に頼らず、自分の手で設計を組み立てるプロセスこそが、設計の本質を最も深く理解できると考えたからです。

実は、この本を書こうと思った大きなきっかけの一つは、自分自身の設計スキルをもっと高めたいという思いにありました。人に伝える形にまとめることで、私自身も設計の奥深さを再確認し、学び直す貴重な機会になりました。仕事の合間や週末の時間の多くを費やし、約半年かけてようやく形にすることができ、今はほっとしています。

本書で紹介した手法は、あくまで「一つの手段」です。実際の現場で設計を適用するのは、簡単なことではありません。大規模な開発では自分一人で完結しない部分も多く、理想の設計をそのまま押し通すことはできません。周囲と歩調を合わせ、期限やコストとのバランスを見極めながら、現実的な落としどころを探っていく……そこには、泥臭い調整や応用力が求められます。

本書のコード例が、そのままあなたの現場で使えるとは私も思っていません。だからこそ「やり方（How）」を丸暗記するのではなく、「なぜそうするのか（Why）」という思考プロセスを身につけていただきたいのです。設計で迷ったとき、本書で触れた「責任」「契約」「依存」という3つの観点が頭の隅にあれば、それが突破口になるはずです。この視点でコードを見直す習慣がつけば、自然とSOLID原則に沿った、現場に即した設計が導き出せるようになると確信しています。

一見複雑に見える設計プロセスも、すべては「価値」を生み出すための手段に過ぎません。設計に取り組む最大の目的は、場当たり的な実装によって失われる「変更のしやすさ」を守ることにあります。いきなりコードを書き始めればその瞬間は速いかもしれませんが、後から優れた設計を付け足すのは極めて困難です。だからこそ、設計の目的を理解し、関係者と合意形成を図りながら、納得感を持って進めていくことが重要です。

もし本書を通じて、C言語での設計の考え方を掴むことができたなら、それは他のあらゆる言語にも通じる強力な武器になります。将来、JavaやC#、Pythonなどを扱う際にも、カプセル化やポリモーフィズムといった機能が「何のためにあるのか」を本質的に理解し、土台として活用できるはずです。それこそが、エンジニアとしての普遍的な成長につながると信じています。

「もっと良くできるはず」という終わりのない試行錯誤こそが、ソフトウェア設計の面白さです。私自身も現場で作り続けながら、設計のレベルをさらに高めていきたいと日々もがいています。

設計という正解のない世界で、本書が皆様のヒントになり、共に成長していくための一助となれば幸いです。
